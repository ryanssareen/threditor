# M1 — Scaffold

## Context

This is the first milestone of the Skin Editor project — a web-based Minecraft skin editor (Next.js 15, R3F, MIT-licensed, zero-infra-cost Phase 1). The goal of M1 is a runnable, deployable shell: a landing page, an `/editor` route with a rotating cube placeholder (to be replaced in M2), the full dependency tree installed with zero peer warnings, and Tailwind v4 token config wired up. M1 locks the package-version compatibility matrix that every downstream milestone will build on; that matrix is the single most important deliverable of this plan, because incorrect peer pairings between React 19 / R3F v9 / drei v10 / three r184 are the failure mode the design doc explicitly flags.

## Overview

Bootstrap a greenfield Next.js 15 App Router project with TypeScript strict mode, install the locked M1 dependency set, wire Tailwind v4 via its PostCSS integration using the CSS-first `@theme` directive (translating the DESIGN.md §2.1 tokens), and render a rotating cube on `/editor` using React Three Fiber v9. Keep the scaffold minimal — do not pre-create directories that M2–M8 will populate. Optimize for `npm install` producing zero peer warnings and `npm run build` succeeding.

8. **Write `app/page.tsx`** — Static Server Component landing:

9. **Write `app/editor/_components/EditorCanvas.tsx`** — the only `'use client'` component in M1:
   ```tsx
   'use client';
   import { Canvas, useFrame } from '@react-three/fiber';
   import { useRef } from 'react';
   import type { Mesh } from 'three';

   function RotatingCube() {
     const ref = useRef<Mesh>(null);
     useFrame((_, delta) => {
       if (!ref.current) return;
       ref.current.rotation.x += delta;
       ref.current.rotation.y += delta * 0.5;
     });
     return (
       <mesh ref={ref}>
         <boxGeometry args={[1, 1, 1]} />
         <meshStandardMaterial color="#00E5FF" />
       </mesh>
     );
   }

   export function EditorCanvas() {
     return (
       <Canvas camera={{ position: [3, 3, 3], fov: 50 }} className="h-dvh w-dvw">
         <ambientLight intensity={0.4} />
         <directionalLight position={[3, 5, 2]} intensity={1.1} />
         <RotatingCube />
       </Canvas>
     );
   }
   ```
   Notes: `useFrame` with `delta` (time-scaled, not `clock.elapsedTime`) keeps rotation consistent across frame rates. `Mesh` imported as `type` (`three` is a heavy module; type-only imports are free). No OrbitControls yet — drei is installed but unused until M2.

10. **Write `app/editor/page.tsx`** — thin wrapper (Server Component is fine; it merely renders a client component):
    ```tsx
    import { EditorCanvas } from './_components/EditorCanvas';
    export default function EditorPage() {
      return <EditorCanvas />;
    }
    ```
    No `'use client'` on the page itself — the client boundary is `EditorCanvas`. R3F v9 handles SSR-safe mounting internally; `next/dynamic` with `ssr: false` is **not** required for a plain `<Canvas>` render path. If future work adds globals that touch `window` at module scope, revisit.

## Known risks

**P1 — `tailwind.config.ts` absence deviates from DESIGN.md §3 file tree.**
DESIGN.md §3's file structure lists `tailwind.config.ts`, but Tailwind v4 is CSS-first and configuring tokens in a `.ts` file fights the framework (v4 killed the JS config; it is supported only as a back-compat affordance via the `@config` directive). This plan puts the DESIGN.md §2.1 tokens into `app/globals.css` via `@theme` instead. The *effect* — `bg-ui-base`, `text-accent`, etc. utilities — is identical. Mitigation: record this deviation in `docs/COMPOUND.md` during the Compound phase so future milestones do not assume a `tailwind.config.ts` exists.

**P1 — Next.js `next@latest` is now 16.2.4, but DESIGN.md pins 15.** If `npm install next` (unpinned) were run, it would install Next 16 and silently skew the project. Mitigation: pin `"next": "15.5.9"` **exactly** (no caret) in `package.json`. Same for `eslint-config-next`.

**P2 — Optional peer warnings from R3F v9.** R3F v9's optional peer deps include `expo`, `expo-asset`, `expo-file-system`, `expo-gl`, `react-native`. npm 10+ should treat these as optional and skip warnings, but some CI/npm versions still emit warnings. Mitigation: pin Node to >=20.9, use `npm` (not `pnpm`/`yarn` for M1). If warnings still appear, document the exact warning text in `docs/COMPOUND.md` and evaluate whether to set `optionalDependencies` or accept the noise.

**P2 — `@types/three` drift.** If `@types/three` is installed with `^` and the latest minor doesn't match `three@0.184`, type errors can appear at build time. Mitigation: pin `@types/three` exactly to `0.184.x`.

**P2 — `useFrame` in React Strict Mode double-mount.** Next.js 15 enables `reactStrictMode: true` by default; R3F v9's migration guide calls out that strict mode exposes prior bugs. For M1's single rotating cube, this is not expected to manifest, but is documented here so M2's player-model work expects it.

**P3 — Node version mismatch on CI vs local.** Next 15.5 requires Node >=20.9. Vercel's default is Node 22 (fine), but local dev on older Node will surface cryptic errors. Mitigation: `"engines": { "node": ">=20.9" }` in `package.json` + optional `.nvmrc` with `20` (deferred).

**P3 — Tailwind v4 + Turbopack CSS HMR edge cases.** Tailwind v4 is fast but a couple of reports exist of stale CSS on first-edit in Turbopack dev mode. Workaround when it appears: hard refresh. Not a scaffold-level issue.

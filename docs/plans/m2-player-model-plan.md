# M2 — Player Model

## Context

This plan is derived from [docs/plans/m2-player-model.md](docs/plans/m2-player-model.md) (round 4 cross-AI dispatch inputs) and [docs/DESIGN.md](docs/DESIGN.md) §6 + §12.5 M2. It replaces M1's rotating-cube placeholder with a full Minecraft player model and locks downstream-critical artifacts (UV constants, camera/breathing/orbit parameters, rim-light color, next/font wiring) that M3-M7 will consume. M2 is the first milestone where getting values *wrong* has cascade cost: every UV coordinate here propagates into raycast paint math (M4), bucket-fill island maps (M5), undo diffs (M6), templates (M7), and OG image rendering (M11). This plan treats those constants as load-bearing contracts and specifies them explicitly rather than deferring to work-phase improvisation.

## Overview

M2 turns `/editor` into a recognizably human Minecraft silhouette. Work spans six surfaces: (1) integrate `next/font` for Geist + JetBrains Mono per Gemini round-4 (A.1, blocking all downstream typography); (2) migrate lint from `next lint` + `.eslintrc.json` to ESLint flat config via the Next.js codemod, which also resolves the 2 low-severity eslint ReDoS advisories; (3) create `lib/three/constants.ts` with camera/breathing/orbit/rim-light pins; (4) create `lib/three/geometry.ts` with 24 UV constants for Classic + Slim via `BoxGeometry.uv`-attribute mutation (ecosystem standard); (5) create `lib/three/PlayerModel.tsx` as the 16-mesh humanoid (8 base + 8 overlay boxes) with a single allocation-free `useFrame` coalescing breathing + micro-orbit; (6) wire variant-toggle local state into `EditorCanvas.tsx` for visual Classic/Slim diff. Default skin is a programmatic placeholder for M2; real Microsoft sample sourcing is pushed to M7 per the inputs file's section C.1.

### World-space box dimensions

1 skin pixel = 0.0625 world units (derived from DESIGN.md §6 head at `boxGeometry(0.5, 0.5, 0.5)` = 8 px/side).

| Part | Base W × H × D (world) | Overlay W × H × D (world) |
|---|---|---|
| Head | 0.5 × 0.5 × 0.5 | 0.5625 × 0.5625 × 0.5625 |
| Body | 0.5 × 0.75 × 0.25 | 0.5625 × 0.8125 × 0.3125 |
| Classic arm | 0.25 × 0.75 × 0.25 | 0.3125 × 0.8125 × 0.3125 |
| Slim arm | 0.1875 × 0.75 × 0.25 | 0.25 × 0.8125 × 0.3125 |
| Leg | 0.25 × 0.75 × 0.25 | 0.3125 × 0.8125 × 0.3125 |

Overlay dimensions = base + 1 pixel on each axis (standard Minecraft skin convention). The overlay material uses `transparent: true` and `alphaTest` or `depthWrite: false` to avoid z-fighting with the base layer.

   export const CAMERA_POSITION = [0, 1.4, 3.2] as const;
   export const CAMERA_LOOK_TARGET = [0, 1.2, 0] as const;
   export const CAMERA_FOV = 32;
   export const IDLE_ORBIT_START_SEC = 0.5;
   export const IDLE_ORBIT_PERIOD_SEC = 9;          // middle of 8-10s range (A.3)
   export const IDLE_ORBIT_AMPLITUDE_RAD = 0.0524;  // 3° in radians
   export const BREATHING_FREQ_HZ = 1.5;
   export const BREATHING_AMPLITUDE = 0.01;

6. **Create `lib/three/geometry.ts`.** Pure-data + pure-function module. Exports:
   - `type SkinVariant = 'classic' | 'slim';`
   - `type UVRect = { x: number; y: number; w: number; h: number };`
   - `type BoxUVs = { top: UVRect; bottom: UVRect; right: UVRect; front: UVRect; left: UVRect; back: UVRect };`
   - `type PlayerUVs = { head: BoxUVs; headOverlay: BoxUVs; body: BoxUVs; bodyOverlay: BoxUVs; rightArm: BoxUVs; rightArmOverlay: BoxUVs; leftArm: BoxUVs; leftArmOverlay: BoxUVs; rightLeg: BoxUVs; rightLegOverlay: BoxUVs; leftLeg: BoxUVs; leftLegOverlay: BoxUVs };`
   - `CLASSIC_UVS: PlayerUVs` — the Classic table above, in type-safe form
   - `SLIM_UVS: PlayerUVs` — Classic with arms swapped for Slim values
   - `BODY_PART_DIMS: { [variant in SkinVariant]: { [part: string]: [W, H, D] } }` — world-space box dimensions
   - `OVERLAY_DIM(base: [W,H,D]): [W,H,D]` — helper: each dim + 2 × 0.0625 (1 pixel on each side)
   - `mapBoxUVs(geometry: BoxGeometry, uvs: BoxUVs, atlasSize: number = 64): void` — mutates `geometry.attributes.uv.array` in place. Face order is three.js BoxGeometry default: `[right, left, top, bottom, front, back]`, each face 4 vertices, vertex order within each face: upper-left `(0,1)`, upper-right `(1,1)`, lower-left `(0,0)`, lower-right `(0,0)`. Per-vertex normalized UV = `(px / 64, 1 - py / 64)`.
   - No `'use client'`. Can be imported from server components and tests.

7. **Create `lib/three/placeholder-skin.ts`.** Pure function:
   ```
   export function createPlaceholderSkinDataURL(variant: SkinVariant): string
   ```
   Generates a 64×64 canvas, fills each body-part UV region (from `CLASSIC_UVS` or `SLIM_UVS`) with a distinct hue so the variant toggle produces a visibly different render. Suggested palette: head = `#8B5A3C` (tan), body = `#3366CC` (blue), arms = `#663399` (purple), legs = `#2E8B57` (green). Return `canvas.toDataURL('image/png')`. No file written to `public/`; texture loader consumes the data URL. No network; runs at module init time.

8. **Create `lib/three/PlayerModel.tsx`.** `'use client'` component. Props: `{ texture: THREE.Texture; variant: SkinVariant }`. Internally:
   - 16 `useMemo`-ed `BoxGeometry` instances (one per part + overlay), each mutated via `mapBoxUVs` with the appropriate `UVs[part]` and `BODY_PART_DIMS[variant][part]` dimensions. Memo key: `[variant]` so variant toggle rebuilds only when variant changes.
   - Overlay materials use `transparent: true, depthWrite: false, alphaTest: 0.01` to prevent z-fighting and preserve correct blend with base.
   - Base materials use `meshStandardMaterial map={texture} transparent` (transparent for the later-added grayscale shader in M8).
   - A single `useRef<THREE.Mesh>` for the head (for breathing) and a `useRef<THREE.PerspectiveCamera>` obtained from `useThree` for orbit.
   - One `useFrame((state) => { ... })` callback coalescing breathing + orbit:
     ```
     const t = state.clock.elapsedTime;
     // breathing — allocation-free
     if (headRef.current) {
       headRef.current.position.y = HEAD_BASE_Y + Math.sin(t * TWO_PI * BREATHING_FREQ_HZ) * BREATHING_AMPLITUDE;
     }
     // micro-orbit — gated on 500ms warm-up
     if (t >= IDLE_ORBIT_START_SEC) {
       const angle = Math.sin((t - IDLE_ORBIT_START_SEC) * TWO_PI / IDLE_ORBIT_PERIOD_SEC) * IDLE_ORBIT_AMPLITUDE_RAD;
       const radius = 3.2;
       state.camera.position.x = Math.sin(angle) * radius;
       state.camera.position.z = Math.cos(angle) * radius;
       state.camera.lookAt(CAMERA_LOOK_TARGET[0], CAMERA_LOOK_TARGET[1], CAMERA_LOOK_TARGET[2]);
     }
     ```
   - No `new Vector3()` inside the callback. `Math.sin/cos` are stack-only, no heap allocation. `camera.lookAt(x, y, z)` uses three.js module-level temp buffers (verified allocation-free).
   - World positions for parts (derived from DESIGN.md §6 pose — head-up, arms-down, legs-down):
     - Head: `[0, 1.4, 0]`
     - Body: `[0, 0.775, 0]`
     - Right arm: `[variant === 'classic' ? -0.375 : -0.34375, 0.775, 0]`
     - Left arm: mirror on X
     - Right leg: `[-0.125, 0, 0]`
     - Left leg: `[0.125, 0, 0]`
     - Overlays share their base's center.

## Known risks

**P1 — Slim UV values — RESOLVED.** Verified against skinview3d source (`bs-community/skinview3d/master/src/model.ts`, `setSkinUVs(box, u, v, w, h, d)` helper) at M2 `/ce:work` start. The slim arm atlas origins are `setSkinUVs(rightArmBox, 40, 16, 3, 12, 4)` and `setSkinUVs(leftArmBox, 32, 48, 3, 12, 4)`; face-by-face algebraic derivation from skinview3d's convention (Right → Front → Left → Back contiguous at `v+d`, Top → Bottom contiguous at `v`) matches every row in this plan's Slim delta table. Packing convention is **contiguous-start at x=40/32 (right/left arm) with unused tail** — confirmed, not centered-with-padding. Captured in `docs/COMPOUND.md` M2 entry during `/ce:compound`.

**P1 — `next/font` → `@theme` var wire-through — RESOLVED.** Verified against M1's committed build output: Tailwind v4 compiles `@theme` vars to `:root` CSS custom properties at **runtime**, not build time. The `.font-sans` utility produces literal `font-family: var(--font-sans)`, so `--font-sans = var(--font-geist), Inter, …` in `@theme` chains correctly through the cascade when next/font sets `--font-geist` on `<body>`. No build-time baking, no fallback global rule needed. The residual risk — portals mounting outside `<body>` losing the `--font-geist` cascade — remains a watch-item for M3+ (dialogs/tooltips); captured in `docs/COMPOUND.md` M2 entry.

**P2 — ESLint flat-config codemod may rewrite `lint` script unexpectedly.** The codemod is a canary (`@next/codemod@canary`), and its output for a brand-new project vs one with an existing `.eslintrc.json` has diverged across patch versions. **Mitigation**: run the codemod once, commit its output immediately as a standalone commit (step 12 commit 1), inspect the generated `eslint.config.mjs` vs the Next.js-documented shape. If it's visibly broken (missing `FlatCompat`, wrong extends, wrong script rewrite), revert the commit and do the migration manually from Next.js docs.

**P2 — Programmatic placeholder skin may look ugly.** Unavoidable — the whole point is debuggability, not aesthetics. **Mitigation**: the palette specified in step 7 has enough saturation difference between parts that variant toggle is obviously visible even if the skin itself is hideous. A `// TODO(M7): replace with Microsoft blank-better.png` comment in `placeholder-skin.ts` prevents it leaking past M7.

**P2 — Single `useFrame` for breathing + orbit risks allocation creep during refactor.** If a future author (M3, likely) adds ref-tracking for hover-highlight to the same callback without thinking about allocations, they'll introduce sawtooth memory. **Mitigation**: add a comment at the top of the `useFrame` callback: `// INVARIANT: zero allocations in this callback. See M1 COMPOUND / M2 risk P2.`

**P3 — Rim-light `#7FD6FF` confidence is low.** ChatGPT round 3 picked the value; Gemini round-4 validation didn't return. **Mitigation**: M2 only pins the constant — it isn't used until M3's hover rim-light (B.5). If M3 visual review shows the color reads wrong against the OLED-dark `#0A0A0A` background, re-dispatch Gemini then. Don't try to re-validate pre-M3.

**P3 — Bundle size may grow past 340 kB First Load.** PlayerModel imports 16 geometries + `useFrame`, plus drei if anything leaks in. M1 baseline was 340 kB; expect M2 in the 360-400 kB range. **Mitigation**: if build report shows >420 kB First Load, bisect: remove PlayerModel and re-check, remove `mapBoxUVs` helper and re-check. If three.js examples leaked in (e.g., from an accidental `import * as THREE`), tree-shake with named imports only.

**P3 — `transparent: true` + overlay meshes may show ordering artifacts on some GPUs.** Overlays at scale 1.125 on head should occlude base head faces. If depth-buffer sorting is unstable on an older GPU, the overlay may z-fight with base. **Mitigation**: `depthWrite: false` on overlay materials + `alphaTest: 0.01` handles most cases. If artifacts persist, add `renderOrder: 1` to overlay meshes and `0` to base meshes.
